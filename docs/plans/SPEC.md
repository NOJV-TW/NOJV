# 可擴展的 Judge Pipeline

## 動機

傳統的 Online Judge (OJ) 只能做純文字的比對，就是拿學生的輸出文字跟標準答案比較是否相同。但對於一個現代化的教育平台來說，這遠遠不夠。老師們想出的題目越來越有創意，有時候需要驗證程式畫出來的圖對不對，甚至還想限制學生不能用某些特定的語法或函式庫。為了滿足各種教學需求，我們需要一個更強大、更有彈性的評測系統。

**一些使用情境：**
> **身為老師**，我希望能自己寫一個檢查腳本 (Checker)，這樣我才能出那種答案不唯一，或是輸出格式比較特別（例如圖片）的題目。

> **身為老師**，我希望能設定一些靜態分析的規則，來強制學生遵守特定的程式碼風格，或是在教學上限制他們使用某些函式庫（例如 C++ 的 STL）。

> **身為課程設計者**，我希望能設計出多階段的作業，一次評測就能從不同面向（例如：正確性、執行效率、程式碼風格）來評估學生的程式。

> **身為學生**，我希望能直接上傳一個包含整個專案的 zip 檔，這樣才能解決那些需要多個檔案才能完成的複雜問題。

> **身為老師**，我希望能提供一個只包含函式宣告的模板，讓學生專心在核心邏輯的實作上。評測時，系統會自動把學生的程式跟老師寫好的主程式、測試碼打包在一起編譯。

## 背景與現況

現在的評測程式碼是寫死的，很難為了不同課程的需求去客製化。我們的目標是，把這個像鐵板一塊的評測流程，改造成一個可以自由組合、擴充的彈性「管線 (Pipeline)」。

## 我們想怎麼做？

我們需要打造一個模組化的評-測系統，核心概念就是這個可擴展的「Judge Pipeline」。簡單來說，每個題目都可以定義自己專屬的一套評測流程，這個流程由一個或多個「階段 (Stage)」串接而成。

### 1. 核心元件

*   **自訂 Checker**：老師可以上傳自己寫的腳本（例如 Python 或 Bash）。這個腳本會拿到學生的輸出、測資的輸入和標準答案等資訊，然後由它來決定要給 AC 還是 WA，甚至可以給出分數。這個腳本會在和學生程式一樣的沙箱環境中執行。
*   **多階段評測**：評測流程會被拆解成一系列的階段，例如 `編譯 (compile)` -> `靜態分析 (static-analysis)` -> `執行 (execute)` -> `檢查 (check)`。老師可以自由新增、移除或調整這些階段的順序。
*   **可擴展的 Pipeline**：整個系統的設計目標，就是要讓未來可以很輕易地加入新的、客製化的階段。這確保平台可以與時俱進，不用為了新需求就整個打掉重練。

### 2. 關鍵功能

*   **語法與靜態分析**：這是一個在程式執行前運作的獨立階段。老師可以設定規則，檢查學生是否用了被禁止的函式、函式庫或語法（例如透過分析 AST）。他們也可以整合市面上主流的 Linter，像是 Python 的 `pylint` 或 JavaScript 的 `eslint`。
*   **支援非標準輸出**：自訂 Checker 可以驗證任何格式的輸出，不管是圖片、JSON/XML，甚至是檔案 I/O 的操作結果。基本上，只要是前一個階段產生的東西，它都能拿來做判斷。
*   **彈性的計分方式**：支援部分給分和子任務 (Subtask) 的評分。最終分數可以由不同階段或不同測資的分數組合而成。對於更複雜的情境，我們甚至可以支援用自訂腳本來算分，例如：遲交的作業要打折。
*   **互動式評測**：建立一個機制，讓學生的程式和評測端的驅動程式可以進行即時的雙向溝通。
*   **自訂評分標準**：允許老師定義加權的評分標準，對應到 Judge Pipeline 中的不同階段。
*   **自訂產物 (Artifacts) 收集**：老師可以定義一份「產物」清單（例如：程式執行後產生的檔案）。評測結束後，這些產物會被打包起來，讓學生可以在結果頁面下載，方便他們除錯。
*   **受控的網路存取**：對於需要連上網路的題目，老師可以開啟受限制的網路功能。這有兩種設定方式：
    *   **防火牆規則**：老師可以定義嚴格的防火牆規則，只允許連線到指定的 IP、網域或 Port。所有的網路活動都會被記錄下來備查。
    *   **Sidecar 服務**：老師可以提供一個容器化的服務（例如一個模擬的 API 或資料庫），這個服務會和學生的程式跑在同一個隔離的網路環境裡，可以透過 `localhost` 來存取。

### 3. 靈活的提交處理

評測系統必須能處理各種不同的作業繳交格式，才能從簡單的單一檔案練習，過渡到更複雜、更貼近真實世界的專案開發。

*   **單一檔案提交**：最傳統的方式，學生只需要上傳一個原始碼檔案。
*   **多檔案專案提交**：學生可以上傳一個 `.zip` 檔，裡面包含完整的專案結構。老師可以在題目中提供 `Makefile` 或自訂的編譯腳本來處理編譯和設定；也可以讓學生自己準備 `Makefile`。
*   **僅實作函式**：對於某些特定的練習，老師可以提供一個只包含函式宣告的模板檔。學生只需要完成函式的實作。評測時，系統會把學生的實作和老師提供的測試程式（例如 `main` 函式）打包在一起編譯成最終的執行檔。

### 4. 一些應用範例

這裡舉幾個例子，說明這個可擴展的 Pipeline 能如何創造出更多元、更有效的程式作業。

#### 範例 1：實作反轉鏈結串列

*   **題目**：給定一個鏈結串列節點的 `struct`，學生需要實作一個函式來反轉這個鏈結串列。
*   **Pipeline 設計**：
    1.  **提交類型**：採用**僅實作函式**的格式。老師提供 `Node` 結構和函式簽名 (例如 `Node* reverse(Node* head);`)。
    2.  **編譯階段**：將學生提交的函式，與老師提供的測試工具一起編譯。這個工具包含一個 `main` 函式，負責建立鏈結串列、呼叫學生的 `reverse` 函式，並遍歷反轉後的結果。
    3.  **檢查階段**：測試工具將反轉後的串列內容印到標準輸出。接著，用一個標準的 `diff` 檢查器，比對這個輸出和預期的正確答案是否一致。

#### 範例 2：串接即時 API 的命令列字典

*   **題目**：學生需要寫一個程式，接收一個單字作為命令列參數，然後去呼叫一個公開的字典 API，並印出這個單字的定義。
*   **Pipeline 設計**：
    1.  **網路存取**：這個題目需要**受控的網路存取**。老師設定**防火牆規則**，只允許對外的 HTTPS 流量連到該字典 API 的特定網域（例如 `api.dictionaryapi.dev`）。所有其他的網路連線都會被阻擋並記錄。
    2.  **檢查階段**：使用**自訂 Checker** 腳本。對於一個給定的單字，Checker 自己也會去呼叫那個字典 API 來取得即時、正確的定義。然後，它會比對學生程式的輸出和它自己拿到的資料，來判斷是否正確，同時也能容忍 API 回應的微小差異。

#### 範例 3：旋轉 BMP 圖片

*   **題目**：要求學生寫一個程式，讀取 `input.bmp`，將圖片順時針旋轉 90 度，並將結果存成 `output.bmp`。
*   **Pipeline 設計**：
    1.  **檢查階段**：這需要一個能做圖片比對的**自訂 Checker**。這個 Checker（例如一個使用 Pillow 函式庫的 Python 腳本）會同時打開學生程式產生的 `output.bmp` 和標準答案 `solution.bmp`。然後，它可以做像素級的比對，或是用感知雜湊 (perceptual hash) 來判斷兩張圖在視覺上是否一致。
    2.  **產物收集**：老師設定這個題目要收集 `output.bmp` 作為**自訂產物**。評測結束後，學生可以直接從結果頁面下載他們程式產生的 `output.bmp`。這對除錯非常有幫助，因為他們能準確看到自己的程式到底產生了什麼東西。

#### 範例 4：根據遲交與資源用量來動態計分

*   **題目**：老師想要套用一個自訂的計分公式，對於遲交或資源用量過多的繳交紀錄給予懲罰。
*   **Pipeline 設計**：
    1.  **核心評測**：先根據程式的正確性進行評測，得到一個 `原始分數`。
    2.  **自訂計分階段**：在 Pipeline 的最後加入一個專門用來算分的階段。這個階段不執行學生程式，而是跑老師提供的一個特殊**計分腳本**。
    3.  **計分腳本邏輯**：這個腳本會收到一個 JSON 物件，裡面包含這次繳交的詳細資訊，例如 `原始分數`、`遲交秒數` 和 `記憶體用量峰值`。腳本接著會根據老師的公式計算出 `最終分數` 並印到標準輸出。
        *   **範例 (遲交懲罰)**：`最終分數 = max(0, 原始分數 * (86400 - 遲交秒數) / 86400)`
        *   **範例 (遲交懲罰)**：`最終分數 = max(0, 原始分數 - 20 * 遲交週數)`
        *   **範例 (記憶體懲罰)**：`最終分數 = 原始分數 * (1.2 - 0.4 * min(記憶體用量MB, 16) / 16)`

## 風險

*   **複雜度**：設計一個有彈性的 Pipeline，肯定比做一個簡單的評測器要複雜得多。
*   **安全性**：自訂的腳本和階段都必須在嚴格控管的沙盒中執行，以防被濫用。而且，這些自訂腳本（例如 Checker）最好不要和學生的程式在同一個環境中執行，因為它們可能需要一些不該讓學生碰到的額外工具。
*   **網路濫用**：只要開放了網路存取，即使是受限制的，也會增加被攻擊的風險，可能被用來做壞事（例如 DDoS）。

## 驗收標準
*   老師可以為題目上傳自訂的 Checker 腳本。
*   老師可以定義一個包含多個階段的 Judge Pipeline（例如：先靜態分析，再執行）。
*   系統能正確抓出並懲罰那些使用了老師所禁止的函式的程式碼。
*   開發者可以在不改動核心評測服務的情況下，為系統新增一個客製化的 Pipeline 階段。
*   系統能正確地編譯並評測以 zip 檔形式繳交的多檔案專案。
*   系統能正確地將「僅實作函式」的繳交與老師提供的程式碼結合併進行評測。
